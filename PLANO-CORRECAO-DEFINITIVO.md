## BUG #1: LEMBRETES RECORRENTES NUNCA DISPARAM (FATAL)

### Localização:
**Workflow:** Lembretes Total Assistente (b3xKlSunpwvC4Vwh)
**Node:** `Get Recurring Reminders`

### O bug:
```
FILTROS ATUAIS (ERRADO):
  next_fire_at >= NOW() + 30 minutos    ← no FUTURO
  next_fire_at <= NOW() - 30 minutos    ← no PASSADO

Isso é IMPOSSÍVEL. Nada pode ser >= futuro E <= passado ao mesmo tempo.
Esta query NUNCA retorna resultados.
LEMBRETES RECORRENTES NUNCA DISPARAM.
```

### Correção:
```
FILTROS CORRETOS (inverter gte/lte):
  next_fire_at <= NOW() + 30 minutos    ← até 30min no futuro
  next_fire_at >= NOW() - 30 minutos    ← desde 30min atrás

Isso cria uma janela de 1 hora ao redor de NOW().
```

### Como aplicar no n8n:
1. Abrir workflow **Lembretes Total Assistente**
2. Abrir node **`Get Recurring Reminders`**
3. No filtro `next_fire_at`:
   - Mudar `gte` → `lte` no campo `{{ $now.plus({ minutes: 30 }) }}`
   - Mudar `lte` → `gte` no campo `{{ $now.minus({ minutes: 30 }) }}`
4. Salvar e ativar

### Comparação com o node que FUNCIONA:
O node `Get Reminders (due now)1` tem os filtros CORRETOS:
```
next_fire_at lte NOW() + 30 min   ✅
next_fire_at gte NOW() - 30 min   ✅
```
Basta copiar a mesma lógica para `Get Recurring Reminders`.

## BUG #2: TABELA `log_total` NÃO EXISTE NO SUPABASE DEV (CRASH A CADA 50s)

### Localização:
**Workflow:** Lembretes Total Assistente
**Node:** `Create a row2`
**Frequência:** A cada 50 segundos (schedule trigger)

### O erro:
```
"Could not find the table 'public.log_total' in the schema cache"
```

O workflow de lembretes roda a cada 50 segundos. Quando um lembrete dispara com sucesso, o node `Create a row2` tenta logar na tabela `log_total` — que existe no banco principal (ldbdtakddxznfridsarn) mas NÃO no banco AI Messages (hkzgttizcfklxfafkzfl) que o workflow está usando.

### Impacto:
- O lembrete DISPARA corretamente (envia WhatsApp)
- Mas CRASHA ao tentar logar
- Isso pode impedir que `Mark as Remembered` rode → lembrete dispara de novo infinitamente
- **5 erros só hoje** por causa disso

### Correção (2 opções):

**Opção A — Criar a tabela no banco AI Messages:**
```sql
CREATE TABLE public.log_total (
  id bigint generated by default as identity primary key,
  user_id text not null,
  acao text not null,
  mensagem text not null,
  categoria text not null default 'geral',
  created_at timestamptz not null default now()
);
```

**Opção B — Apontar o node para o banco principal:**
No node `Create a row2`, trocar a credencial Supabase para a do banco principal (ldbdtakddxznfridsarn).

## BUG #5: JANELA DE CAPTURA DE LEMBRETES MUITO ESTREITA

### Localização:
**Node:** `Get Events (in 30 min)` — para tipo "lembrete"

### O problema:
```
start_event > NOW()
start_event < NOW() + 2 minutos    ← janela de só 2 minutos!
```

O schedule roda a cada 50 segundos. Uma janela de 2 minutos significa que um lembrete tem no máximo 2 chances de ser capturado. Se o schedule falhar 2x seguidas (ex: por erro do log_total), o lembrete é perdido para sempre.

### Comparação:
- `Get Events (in 30 min)1` — para tipo "compromisso": janela de **30 minutos** ✅
- `Get Events (in 30 min)` — para tipo "lembrete": janela de **2 minutos** ❌

### Correção:
Mudar `{{ $now.plus({ minutes: 2 }) }}` para `{{ $now.plus({ minutes: 5 }) }}` no node `Get Events (in 30 min)`.

E garantir que `Mark as Remembered` rode ANTES do log_total (pra não perder o lembrete se o log falhar).

## BUG #8: EDIÇÃO DE EVENTOS VIA AI AGENT — LENTIDÃO

### Causa raiz: Redis ENOTFOUND (Bug #4)

O AI Agent usa Redis para carregar contexto. Com Redis down:
1. Node `Redis1` tenta conectar → espera timeout (~17min nos piores casos)
2. Toda operação de edição/exclusão herda essa latência
3. Por isso editar evento leva 45-68s ao invés de ~5s

### Correção:
Resolver Bug #4 (Redis) resolve automaticamente a lentidão.

### Medida adicional — Botão de exclusão ao criar evento:

Quando a IA cria um evento, enviar mensagem interativa com botões:

```json
{
  "type": "interactive",
  "interactive": {
    "type": "button",
    "body": {
      "text": "✅ Evento agendado!\n📅 Aula de Inglês\n🔁 Toda terça e quinta às 19h"
    },
    "action": {
      "buttons": [
        {
          "type": "reply",
          "reply": {
            "id": "delete_event_{{EVENT_ID}}",
            "title": "🗑️ Excluir"
          }
        },
        {
          "type": "reply",
          "reply": {
            "id": "edit_event_{{EVENT_ID}}",
            "title": "✏️ Editar"
          }
        }
      ]
    }
  }
}
```

**Implementação no n8n:**
1. No workflow que envia confirmação de evento, trocar `type: text` por `type: interactive`
2. Incluir o ID do evento no botão
3. No webhook principal, adicionar handler para `type: interactive` → `button_reply`
4. Se `id` começa com `delete_event_` → excluir direto pelo ID
5. Se `id` começa com `edit_event_` → perguntar o que quer mudar

**Benefício:** Exclusão em 1 clique, sem busca, sem ambiguidade, sem timeout.
